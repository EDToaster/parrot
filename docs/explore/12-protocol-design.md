## 1. Design Philosophy

This protocol draws inspiration from **Playwright's action-consequence model**: when you click a button in Playwright, you can wait for the navigation, the network response, or the DOM change that results. Similarly, when you interact with a block in Minecraft, you want to wait for the chest GUI to open and get its contents back in the same response.

**Core principles:**

- **Action-consequence coupling**: Every action returns not just "ok" but the observable consequences of that action (screens opening, blocks changing, entities spawning).
- **Tick-aware timing**: Consequences are collected over a configurable tick window after the action executes, because Minecraft effects are not instantaneous.
- **Concurrent-safe**: Multiple requests can be in-flight; each is correlated by a unique request ID.
- **Dual-mode observation**: Pull (request-response with consequences) and push (event subscriptions for ongoing monitoring).

## 2. Connection Lifecycle

### 2.1 Connection Establishment

```
MCP Server                          Minecraft Mod (Netty)
    |                                       |
    |--- HTTP Upgrade (WebSocket) --------->|
    |    Origin: localhost                   |
    |<-- 101 Switching Protocols -----------|
    |                                       |
    |--- hello (with auth token) ---------->|
    |<-- hello_ack (with capabilities) -----|
    |                                       |
    |    [connection is now active]          |
```

### 2.2 Handshake: `hello` / `hello_ack`

After the WebSocket connection is established, the client MUST send a `hello` message within 5 seconds or the server closes the connection.

**Client sends:**
```json
{
  "type": "hello",
  "id": "1",
  "protocol_version": 1,
  "auth_token": "a1b2c3d4e5f6..."
}
```

**Server responds:**
```json
{
  "type": "hello_ack",
  "id": "1",
  "protocol_version": 1,
  "minecraft_version": "1.21.5",
  "mod_loader": "fabric",
  "mod_version": "0.1.0",
  "capabilities": [
    "actions",
    "queries",
    "events",
    "commands"
  ],
  "tick_rate": 20,
  "server_type": "integrated"
}
```

The `auth_token` is a shared secret generated by the mod on server start and written to a known file path (e.g., `~/.minecraft-mcp/auth-token`). The MCP server reads this file to authenticate.

### 2.3 Heartbeat

Either side may send a `ping`; the other side MUST respond with `pong`.

```json
{ "type": "ping", "id": "42", "timestamp": 1710000000000 }
```
```json
{ "type": "pong", "id": "42", "timestamp": 1710000000000 }
```

The mod sends a `ping` every 30 seconds. If no `pong` is received within 10 seconds, the connection is considered dead and is closed. The MCP server should implement the same logic.

### 2.4 Graceful Disconnect

```json
{ "type": "goodbye", "id": "99", "reason": "client_shutdown" }
```

The other side acknowledges with a `goodbye_ack` and closes the WebSocket.

## 3. Message Framing

All messages are UTF-8 JSON text frames. Binary frames are not used.

### 3.1 Base Message Structure

Every message has a `type` field. Messages sent by the client that expect a response also carry an `id` field.

```typescript
// Client -> Server (requests)
{
  "type": string,       // message type
  "id": string,         // unique request ID (client-generated, UUID or incrementing)
  ...                    // type-specific fields
}

// Server -> Client (responses)
{
  "type": string,       // "<original_type>_result" or "error"
  "id": string,         // echoed request ID
  ...                    // type-specific fields
}

// Server -> Client (push notifications)
{
  "type": "event",
  "subscription_id": string,
  ...                    // event-specific fields
}
```

### 3.2 Message Types Summary

| Type | Direction | Description |
|------|-----------|-------------|
| `hello` | C -> S | Authentication handshake |
| `hello_ack` | S -> C | Handshake acknowledgment |
| `ping` / `pong` | Bidirectional | Keepalive |
| `goodbye` | Bidirectional | Graceful disconnect |
| `action` | C -> S | Perform an action, return consequences |
| `action_result` | S -> C | Action result with collected consequences |
| `query` | C -> S | Read game state (no side effects) |
| `query_result` | S -> C | Query response |
| `command` | C -> S | Execute a Minecraft `/command` |
| `command_result` | S -> C | Command output |
| `subscribe` | C -> S | Start receiving push events |
| `subscribe_ack` | S -> C | Subscription confirmed |
| `unsubscribe` | C -> S | Stop receiving push events |
| `unsubscribe_ack` | S -> C | Unsubscription confirmed |
| `event` | S -> C | Push event notification |
| `error` | S -> C | Error response to any request |

## 4. Request/Response Protocol

### 4.1 Request IDs

Every client request carries a unique `id` string. The server echoes this `id` in the response. This enables:

- **Concurrent requests**: Multiple requests can be in-flight simultaneously.
- **Client-side correlation**: The MCP server can match responses to the MCP tool calls that originated them.
- **Timeout tracking**: The client can time out individual requests independently.

IDs are opaque strings. Recommended format: incrementing integers as strings (`"1"`, `"2"`, ...) or UUIDs.

### 4.2 Request Ordering

Requests are processed in the order they arrive in the command queue. However, because actions block for their consequence window, a long-running action will delay subsequent requests. The mod processes one action per tick drain cycle to maintain game responsiveness.

**Important**: Queries (read-only) are fast and do not have a consequence window. They execute and return immediately within the same tick. Actions take at minimum `consequence_wait` ticks.

### 4.3 Concurrency Model

```
Tick N:   Drain queue -> Execute action A1 (interact_block)
Tick N+1: Collecting consequences for A1...
Tick N+2: Collecting consequences for A1...
Tick N+3: Collecting consequences for A1...
Tick N+4: Collecting consequences for A1...
Tick N+5: Consequence window closes, send A1 result. Execute query Q1 (immediate).
Tick N+5: Drain queue -> Execute action A2...
```

Queries can be interleaved between action consequence windows. Actions are serialized to avoid ambiguity about which action caused which consequence.

## 5. Consequence Collection Model

This is the core innovation of the protocol. Rather than returning a bare "ok" when an action is performed, the mod **waits and observes** what happens in the game as a result.

### 5.1 How It Works

```
1. Client sends:   action { method: "interact_block", ... }
2. Mod executes:   Right-click the block at (x, y, z) on main thread
3. Mod waits:      Listen for game events for N ticks (default: 5 = 250ms)
4. Mod collects:   Screen opened? Block changed? Entity spawned? Sound played?
5. Mod returns:    action_result { consequences: [...collected events...] }
```

### 5.2 Action Request Format

```json
{
  "type": "action",
  "id": "7",
  "method": "interact_block",
  "params": {
    "position": { "x": 100, "y": 64, "z": -200 },
    "face": "top",
    "hand": "main_hand"
  },
  "consequence_wait": 5
}
```

**Fields:**
- `method`: The action to perform (see action catalog below).
- `params`: Method-specific parameters.
- `consequence_wait`: Number of ticks to wait for consequences after the action executes. Default: 5 (250ms). Set to 0 for fire-and-forget. Max: 100 (5 seconds).

### 5.3 Action Result Format

```json
{
  "type": "action_result",
  "id": "7",
  "success": true,
  "tick": 48231,
  "result": {
    "interaction_result": "success"
  },
  "consequences": [
    {
      "type": "screen_opened",
      "tick": 48232,
      "screen_type": "generic_container",
      "title": "Chest",
      "slots": [
        { "index": 0, "item": "minecraft:diamond", "count": 3, "nbt": {} },
        { "index": 1, "item": "minecraft:iron_ingot", "count": 64, "nbt": {} },
        { "index": 2, "item": "minecraft:air", "count": 0 }
      ],
      "slot_count": 27
    }
  ]
}
```

**Fields:**
- `success`: Whether the action itself executed without error.
- `tick`: The game tick when the action was executed.
- `result`: Immediate return value of the action (e.g., interaction result enum).
- `consequences`: Array of observed game events that occurred during the consequence window, each tagged with the tick it occurred on.

### 5.4 Consequence Event Types

The mod installs temporary event listeners during the consequence window. Events collected:

| Consequence Type | Description | Key Fields |
|-----------------|-------------|------------|
| `screen_opened` | A GUI screen opened | `screen_type`, `title`, `slots[]`, `slot_count` |
| `screen_closed` | A GUI screen closed | `screen_type` |
| `block_changed` | A block state changed near the action | `position`, `old_state`, `new_state` |
| `block_entity_changed` | A block entity's data changed | `position`, `block_type`, `data` |
| `entity_spawned` | An entity appeared | `entity_type`, `entity_id`, `position` |
| `entity_removed` | An entity was removed | `entity_type`, `entity_id` |
| `sound_played` | A sound effect played | `sound_id`, `position`, `volume` |
| `particle_spawned` | Particles appeared | `particle_type`, `position`, `count` |
| `chat_message` | A chat/system message appeared | `message`, `sender`, `is_system` |
| `inventory_changed` | Player inventory changed | `slot`, `old_item`, `new_item` |
| `status_effect_added` | Player gained a status effect | `effect`, `duration`, `amplifier` |
| `redstone_update` | Redstone signal changed | `position`, `old_power`, `new_power` |

### 5.5 Consequence Filtering

To reduce noise, clients can specify which consequence types they care about:

```json
{
  "type": "action",
  "id": "8",
  "method": "interact_block",
  "params": { "position": { "x": 100, "y": 64, "z": -200 } },
  "consequence_wait": 5,
  "consequence_filter": ["screen_opened", "screen_closed"]
}
```

If `consequence_filter` is omitted, all consequence types are collected.

### 5.6 Spatial Filtering

Consequences are filtered spatially by default to avoid picking up unrelated world events. Block changes and entity events are only collected within a configurable radius of the action's target position (default: 8 blocks). Screen events, chat messages, and inventory changes are always collected (they are inherently player-scoped).

### 5.7 Action Catalog

| Method | Description | Key Params |
|--------|-------------|------------|
| `interact_block` | Right-click a block | `position`, `face`, `hand` |
| `attack_block` | Left-click / start breaking a block | `position`, `face` |
| `place_block` | Place a block from inventory | `position`, `face`, `slot` |
| `use_item` | Use the held item | `hand` |
| `interact_entity` | Right-click an entity | `entity_id`, `hand` |
| `attack_entity` | Attack an entity | `entity_id` |
| `click_slot` | Click a slot in an open container | `slot_index`, `button`, `click_type` |
| `close_screen` | Close the current screen | (none) |
| `drop_item` | Drop item from inventory | `slot`, `drop_all` |
| `swap_hands` | Swap main/offhand | (none) |
| `set_held_slot` | Change hotbar selection | `slot` (0-8) |
| `move_to` | Move player to position | `position`, `sprint` |
| `look_at` | Change player look direction | `position` or `yaw`/`pitch` |
| `send_chat` | Send a chat message | `message` |

## 6. Query Protocol

Queries read game state without side effects. They execute immediately (no consequence window).

### 6.1 Query Request

```json
{
  "type": "query",
  "id": "12",
  "method": "get_block",
  "params": {
    "position": { "x": 100, "y": 64, "z": -200 }
  }
}
```

### 6.2 Query Result

```json
{
  "type": "query_result",
  "id": "12",
  "tick": 48235,
  "result": {
    "block": "minecraft:chest",
    "state": { "facing": "north", "type": "single", "waterlogged": false },
    "block_entity": {
      "type": "minecraft:chest",
      "custom_name": null,
      "items": "use interact_block to open and read contents"
    }
  }
}
```

### 6.3 Query Catalog

| Method | Description | Key Params |
|--------|-------------|------------|
| `get_block` | Get block state at position | `position` |
| `get_blocks` | Get block states in a region | `from`, `to` (corners) |
| `get_entity` | Get entity data by ID | `entity_id` |
| `get_entities` | Get entities in area | `position`, `radius`, `type_filter` |
| `get_player` | Get player state | (none) — returns position, health, hunger, inventory, effects |
| `get_inventory` | Get player or open container inventory | `target` ("player" or "screen") |
| `get_screen` | Get current open screen info | (none) |
| `get_world` | Get world info | (none) — returns dimension, time, weather, difficulty |
| `get_registries` | List registered blocks/items/entities | `registry` (e.g., "block", "item") |
| `get_loaded_mods` | List loaded mods | (none) |
| `get_nearby_blocks` | Scan for specific block types | `position`, `radius`, `block_filter` |

## 8. Event Subscription System

For ongoing monitoring without polling. The client subscribes to event types and receives push notifications.

### 8.1 Subscribe

```json
{
  "type": "subscribe",
  "id": "20",
  "event_types": ["chat_message", "entity_spawned"],
  "filter": {
    "position": { "x": 100, "y": 64, "z": -200 },
    "radius": 32
  }
}
```

### 8.2 Subscribe Acknowledgment

```json
{
  "type": "subscribe_ack",
  "id": "20",
  "subscription_id": "sub-1"
}
```

### 8.3 Push Event

```json
{
  "type": "event",
  "subscription_id": "sub-1",
  "tick": 48300,
  "event_type": "chat_message",
  "data": {
    "message": "Hello world!",
    "sender": "Steve",
    "is_system": false
  }
}
```

### 8.4 Unsubscribe

```json
{
  "type": "unsubscribe",
  "id": "21",
  "subscription_id": "sub-1"
}
```

```json
{
  "type": "unsubscribe_ack",
  "id": "21",
  "subscription_id": "sub-1"
}
```

### 8.5 Subscribable Event Types

Same types as consequence events (section 5.4), plus:

| Event Type | Description | Key Fields |
|-----------|-------------|------------|
| `player_health_changed` | Health/hunger changed | `health`, `food`, `saturation` |
| `player_moved` | Player position changed | `position`, `yaw`, `pitch` |
| `dimension_changed` | Player changed dimension | `from`, `to` |
| `death` | Player died | `message` |
| `respawn` | Player respawned | `position` |
| `block_break_progress` | Block being broken | `position`, `progress` (0.0-1.0) |
| `advancement` | Player earned advancement | `advancement_id`, `title` |

### 8.6 Subscription vs. Consequence Collection

| Aspect | Consequences | Subscriptions |
|--------|-------------|---------------|
| Trigger | Action execution | Ongoing |
| Duration | N ticks after action | Until unsubscribed |
| Scope | Spatially near action | Configurable filter |
| Delivery | In action_result | Separate event messages |
| Use case | "What happened when I clicked?" | "Tell me whenever a creeper spawns" |

## 9. Error Handling

### 9.1 Error Response Format

Any request can produce an error response instead of the expected result:

```json
{
  "type": "error",
  "id": "7",
  "code": "BLOCK_OUT_OF_RANGE",
  "message": "Target block at (1000000, 64, -200) is not in a loaded chunk",
  "details": {
    "position": { "x": 1000000, "y": 64, "z": -200 },
    "loaded_range": 128
  }
}
```

### 9.2 Error Codes

| Code | Description |
|------|-------------|
| `INVALID_REQUEST` | Malformed JSON or missing required fields |
| `UNKNOWN_METHOD` | Unrecognized action or query method |
| `INVALID_PARAMS` | Invalid parameter values |
| `AUTH_FAILED` | Authentication token is invalid |
| `NOT_AUTHENTICATED` | Request sent before hello handshake |
| `BLOCK_OUT_OF_RANGE` | Target block is not in a loaded chunk |
| `ENTITY_NOT_FOUND` | Entity with given ID does not exist |
| `NO_SCREEN_OPEN` | Action requires an open screen but none is open |
| `SCREEN_ALREADY_OPEN` | Action would open a screen but one is already open |
| `INVALID_SLOT` | Slot index out of range |
| `COMMAND_FAILED` | Minecraft command execution failed |
| `PERMISSION_DENIED` | Player lacks permission for the action |
| `TIMEOUT` | Consequence collection timed out (mod-side issue) |
| `INTERNAL_ERROR` | Unexpected mod-side error |
| `RATE_LIMITED` | Too many requests |
| `PROTOCOL_ERROR` | Protocol violation (e.g., wrong protocol_version) |

### 9.3 Timeout Handling

**Client-side timeout**: The MCP server should implement its own timeout per request (recommended: 10 seconds). If no response arrives within this window, the client can:
1. Send a `cancel` message: `{ "type": "cancel", "id": "7" }`
2. Assume the request failed and surface the error to the MCP caller.

**Server-side timeout**: The consequence window is bounded by `consequence_wait` ticks. If the mod's tick loop stalls (e.g., server lag), the mod uses a wall-clock fallback: if more than `consequence_wait * 100ms` has elapsed (2x the expected duration), the consequence window closes regardless.

**Consequence "empty" is not an error**: If no consequences are observed during the wait window, the response still succeeds — the `consequences` array is simply empty. This is expected for actions like placing a block in open air (no screen opens, no special events).

## 11. Design Decisions & Rationale

### 11.1 Why Tick-Based Consequence Windows (Not Event-Driven Completion)

**Alternative considered:** Wait for a specific expected event (e.g., "wait for screen_opened") like Playwright's `waitForNavigation()`.

**Why we chose tick-based:** Minecraft actions have unpredictable consequences. Right-clicking a block might open a chest, activate a redstone circuit, trigger a mod's custom handler, or do nothing. We cannot predict what events to wait for. A fixed tick window collects *whatever happens*, making the protocol robust to unknown mod behaviors.

**Tradeoff:** Slightly higher latency (always waits the full window) vs. robustness. Mitigated by keeping the default window short (5 ticks = 250ms) and making it configurable per-request.

### 11.2 Why Serialized Actions (Not Parallel)

Actions are executed one at a time because:
1. **Consequence attribution**: If two actions execute simultaneously, it's ambiguous which action caused which events.
2. **Game state consistency**: Minecraft's game loop is single-threaded. Interleaving actions within a tick would be artificial.
3. **Debugging clarity**: For the mod debugging use case, sequential actions with clear consequences are more useful than parallel execution.

Queries are not serialized — they can be batched and executed within the same tick since they have no side effects.

### 11.3 Why Separate Action/Query/Command Types

- **Actions** have consequences and modify game state. They use the consequence collection system.
- **Queries** are read-only and return immediately. No consequence window needed.
- **Commands** are a raw escape hatch. They can have consequences but are less structured.

Separating them allows the mod to optimize execution (queries are cheap, actions need the event listener infrastructure) and the MCP server to set appropriate timeouts.

### 11.4 Why Not Use JSON-RPC

JSON-RPC 2.0 was considered but rejected because:
1. Our protocol needs push notifications (events), which JSON-RPC doesn't natively support.
2. The `consequences` array is a protocol-specific concept that doesn't map cleanly to JSON-RPC's simple result/error model.
3. JSON-RPC's batch mode would add complexity without benefit (we serialize actions anyway).

However, the protocol is *inspired by* JSON-RPC: request IDs, method names, params objects, and error codes follow similar patterns for familiarity.

### 11.5 Auth Token File Approach

The mod generates a random token on each server start and writes it to a known file. This is the same approach used by:
- Jupyter Notebook (writes token to console/file)
- Chrome DevTools Protocol (writes WebSocket URL to stderr)
- GDMC HTTP (uses localhost binding without auth)

We add the token on top of localhost-only binding for defense in depth against cross-site WebSocket hijacking.

## 12. Implementation Notes for Mod Developers

### 12.1 Consequence Collection Architecture

```
Action executed on Tick N:
  1. Install temporary event listeners for all consequence types
  2. Store listeners in a ConsequenceCollector with deadline = N + consequence_wait
  3. Each tick, ConsequenceCollector checks if deadline reached
  4. When deadline reached:
     a. Uninstall all temporary listeners
     b. Gather collected events
     c. Complete the CompletableFuture with action_result + consequences
```

### 12.2 Event Listener Strategy

**Fabric:** Use `ScreenEvents.AFTER_INIT`, `ServerTickEvents`, and direct world observation. For block changes, use a Mixin on `ServerLevel.setBlock()` or poll a small region around the action target.

**NeoForge:** Use `ScreenEvent.Opening`, `BlockEvent`, `EntityEvent`, and `ServerTickEvent`. The event bus makes this straightforward.

**Common module:** The ConsequenceCollector, event types, and JSON serialization are loader-independent. Only the event listener registration is loader-specific.

### 12.3 Block Change Detection

Block changes don't have a universal event on both loaders. Two approaches:

1. **Snapshot comparison**: Capture block states in a radius around the action target before execution, then compare after the consequence window. Simple and loader-independent.
2. **Event hooking**: Use NeoForge's `BlockEvent` or Fabric Mixins on `Level.setBlock()`. More precise but loader-specific.

**Recommendation:** Use snapshot comparison in the common module as the primary approach, with event hooking as an optimization in loader-specific modules.

### 12.4 Message Size Limits

- Maximum message size: 1 MB (configurable)
- If a query result would exceed this (e.g., `get_blocks` on a huge region), return a `RESULT_TOO_LARGE` error with a suggestion to reduce the query scope.
- Slot data in `screen_opened` consequences should include all slots, but truncate NBT data for items beyond a configurable depth.

## Appendix B: Future Considerations

- **Binary protocol**: If JSON serialization becomes a bottleneck for large world queries, consider MessagePack or a custom binary format. Netty pipelines make this easy to swap.
- **Multiplayer**: Current design assumes single-player or a single controlling client. For multiplayer, add player targeting to actions and player filtering to subscriptions.
- **Scripting integration**: If a scripting engine (e.g., Rhino/JavaScript) is added, a `script` message type could execute arbitrary scripts with consequence collection.
- **Replay/recording**: The event stream could be recorded for debugging session replay.
